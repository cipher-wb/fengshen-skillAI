# 所有 ID 必须 ≤ int32 max — Newtonsoft.Json 静默截断坑

## 决策
✅ **入库 v2.3.1（hotfix）**

## 现象（用户实测 v2.3 报错）

Unity 控制台：
```
JsonReaderException: Could not convert to integer: 30142003 01
[NodeEditor] Error: [SkillGraph_30142003_残影火影分身-[SkillTagsConfig:-1280766994]] 反序列化数据失败!
[NodeEditor] Error: [SkillGraph_30142003_残影火影分身-[SkillTagsConfig:-1280766995]] 反序列化数据失败!
[NodeEditor] Error: [SkillGraph_30142003_残影火影分身-修改技能参数值 [SkillEffectConfig:32003011]] 反序列化数据失败!
[NodeEditor] Error: [SkillGraph_30142003_残影火影分身-修改技能参数值 [SkillEffectConfig:32003010]] 反序列化数据失败!
[ID异常] 修改技能参数值 [SkillEffectConfig:0], ID: 0, ConfigID: 0
```

## 根因
v2.3 PostMortem #017 的 tag ID 命名约定 `skill_id*100+offset`：
```
30142003 * 100 + 1 = 3014200301
30142003 * 100 + 2 = 3014200302
```

这两个值都 **> int32 max (2147483647)**：
```
3014200301 - 2^32 = -1280766995  ← 错误日志里看到的负数
3014200302 - 2^32 = -1280766994  ← 同上
```

后果链：
1. Newtonsoft.Json 把 ConfigJson 字符串里的 3014200301 反序列化为 int32 → 溢出为 -1280766995
2. SkillEditor 用这个负数找 SkillTagsConfig → 找不到（因为节点 ID=正数 3014200301）
3. 整个 ConfigJson 节点反序列化失败 → 节点 ID 全部回退为 0
4. 多个节点 ID=0 → "ID重复" 报错连环爆

## 修法（已落地）

### 1. tag ID 改用 `skill_id * 10 + offset`
```
v2.3:    3014200301 / 3014200302  ❌ 溢出
v2.3.1:  301420030  / 301420031   ✓ < int32 max (2.1 亿)
```
- 30142003 * 10 = 301420030
- 30142003 * 10 + 1 = 301420031
- 仍保留 "可一眼识别归属技能" 的能力（前 8 位 = skill_id）

### 2. 编译器加 INT32_MAX 防御
```python
INT32_MAX = 2_147_483_647

def _check_int32(value, label):
    if value > INT32_MAX or value < -INT32_MAX - 1:
        raise CompileError(f"{label} ID={value} 溢出 int32...")

def validate_ir(ir):
    schema_validate(ir)
    _check_int32(skill_id, "meta.skill_id")
    for tag in ir.get("tags", []) or []:
        _check_int32(tag["id"], f"tags[{tag['name']}]")
    _walk(flow)  # 递归扫 record_position_to_tag / target_position_tag / second_stage_skill_id
```

任何 IR 写超过 int32 max 的 ID，编译会**直接报错**而不是产出会爆的 JSON。

## 弊
- v2.3 文档里写的 `skill_id*100+offset` 命名约定是错的，要改 docs/易错点速查.md 撤回
- 项目里所有现有 SkillGraph_*.json 没有任何 int32 越界（都是 7-8 位 ID），但若策划手写超长 tag ID 仍会遇到（编译器现在会拦下）

## 噪音风险
低 — 这是数值正确性硬错误（Newtonsoft.Json 行为是 Unity/.NET 标配，无法改），编译期防御是唯一选择

## 反思 — 为什么没在编译期发现？
1. **JSON 输出阶段没做 int32 越界扫描**——所有数字字段都是 Python int（无界），写到 JSON 也是无损的（JSON 数字无大小限制）
2. **Lint 没检查 int32**——Lint 检查的是节点拓扑/语义，不是数值范围
3. **Schema 没限制 max**——`{ "type": "integer" }` 默认无上界

**根本预防措施**：所有 ID 类字段在 Schema 里加 `"maximum": 2147483647`。但工作量大且滞后，本次先在 validate_ir 里硬代码兜底。
