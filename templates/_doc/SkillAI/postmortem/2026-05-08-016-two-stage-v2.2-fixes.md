# 两段式技能 v2.2 — Unity 实测后的 3 个真坑

## 决策
✅ **入库 v2.2**

## 现象（用户实测 v2.1 后反馈 3 个问题）

> 1. 你技能CD不对，都太长了，导致我返回的技能进入CD了，没有办法触发2段效果
> 2. 你切换的槽位不对，你切换的是神通槽，但实际上你应该要获取当前装备技能的槽位去替换
> 3. 二段回来不对，人物并没有回来，应该是坐标不对，1段应该要存一下位移前的坐标才行

## 根因（3 条都是 v2.1 的 PoC 简化引爆）

### #1 第二段 CD 太长
v2.1 `_split_two_stage` 深拷贝主 IR 作为 second_ir，**继承了主技能的 cd_frames=360**。30221002 实测对比：CdTime=18 帧（≈0.6s），而 30221000 主技能 CdTime=210。第二段是"触发型子技能"（切槽位临时上挂），不是独立 CD 技能。

### #2 切槽位写死
v2.1 用 sub_type='神通' 推断 slot=4（TSST_SHEN_TONG）写到 44014633 [4] 槽位参数。但实际：技能可能装备在不同槽位（不同流派 / 不同技能装备组合）。30221000 实战用的是 **TSET_GET_SKILL_SLOT_TYPE([41/PT=5, 0])** = 取 caster 当前初始技能实例的槽位（动态）。

### #3 第二段位置不对
v2.1 `expand_return_to_clone` 用 `attr:位置X / attr:位置Y` 当目标点，**这是 caster 当前位置**（已位移过）。30221000 实战用的是：
1. 在主技能里召唤残影时**记录残影 entity ID 到 SkillTag**
2. 第二段 GET_ENTITY_ATTR_VALUE 从 SkillTag 取残影 entity → 读其 X/Y

我的简化方案：直接在主技能 summon_clone 时记录 caster.X→tag, caster.Y→tag+1，第二段 return_to_clone 用 PT=3 SKILL_PARAM 直接读这两个 tag 当目标坐标。

## v2.2 修法（已落地）

### 1. CD 修复
```python
# _split_two_stage(...)
main_bd = ir["skill"].get("base_duration", 30)
second_cd = cfg.get("second_stage_cd_frames", main_bd)   # 默认 = bd（不继承主 CD）
second_ir["skill"]["cd_frames"] = second_cd
# 收紧 bd / cast 满足时长四规则
```
- 30142004 v2.2: cd=30（≈1秒），原 v2.1 cd=360
- 30142003 IR 显式写 `second_stage_cd_frames: 30`

### 2. 动态槽位
```python
def _make_get_skill_slot_node(alloc, ctx) -> Node:
    """生成 TSET_GET_SKILL_SLOT_TYPE 取 caster 当前初始技能槽位
       Params=[41 PT=5, 0 PT=0]"""

# expand_two_stage_skill(...)
if swap_slot == "DYNAMIC":  # 默认值
    slot_node = _make_get_skill_slot_node(alloc, ctx)
    slot_param = TParam(value=slot_node.config_id, param_type=2)  # PT=2 effect_return
```
- IR `swap_slot` 默认改为 `'动态'`
- 编译产物：44014633 Param[4] 由 V=4/PT=0 改为 V=32003015/PT=2（指向 GET_SKILL_SLOT_TYPE 节点）

### 3. 位置记录（双 SkillTag 方案）
```python
# expand_summon_clone 当 record_position_to_tag=N 时，自动生成：
#   GET_ATTR(caster, 位置X=59) ──effect_return──→ MODIFY_TAG(tag=N, value=PT=2)
#   GET_ATTR(caster, 位置Y=60) ──effect_return──→ MODIFY_TAG(tag=N+1, value=PT=2)
#   ORDER 包裹 [position-record, summon] 作为代表节点
```
- 30142003 主 JSON 多出 5 个节点：1 wrap_order + 1 inner_order + 2 modify_tag + 2 get_attr（位置X/Y）+ 4 边
- 节点数 14 → 21（与 v2.0 21 节点持平）
- 30142004 return_to_clone 用 `tag:320200` (X) / `tag:320201` (Y) 取回

## 利
- 三个核心 PoC 简化全部升级为生产级实现
- 与 30221000 真实实战范式一致（动态槽位 / SkillTag 存读位置）
- IR 层暴露 `second_stage_cd_frames` + `swap_slot=动态` 默认，新写两段技能不会再踩相同坑

## 弊
- 节点数从 14 增到 21（+50%），但与 v2.0 相同（21 节点），不算膨胀
- 位置记录用 caster.X/Y 而非 clone.X/Y——若 summon_clone 的 position 字段不是 "当前位置"（如朝向偏移、或 indicator 位置），会失真。30142003 用 `position: 当前位置` 没问题，未来需要更复杂的 clone 位置时再扩展

## 噪音风险
低 — 三个修复都对应明确的 Unity 实测 bug，不是凭空猜测

## 历史教训打包
| 版本 | 谁简化了 | 后果 |
|---|---|---|
| v2.0 | second_stage 嵌套 ORDER | 按键不响应（被 v2.1 重做） |
| v2.1 | second CD 继承 / 槽位写死 / 位置=caster.attr | 三处实测 bug（被 v2.2 重做） |
| v2.2 | 位置只记 caster 不记 clone | 留待后续如有偏移需求再扩展 |

**反思**：IR PoC 简化时要更激进地参考"已上线技能"的真实数据形态。例如本次 30221000/30221002 是现成的两段技能，**早一点对照它的 ConfigJson 取值范围（cd=18 / 槽位=GET / 位置=GET_TAG）**，可以一次写对而不是分 3 个版本爬。
