# PostMortem #2026-05-09-023 — TCO 操作符错配 + E021 槽位无消费 lint

## 触发

用户在 v0.3 旋转扩张子弹圈模板 Unity 实测后反馈 4 条：
1. 有些线没有连
2. 我改了参数无效
3. damagePerHit 应该没用（伤害是 BulletConfig 命中链管的）
4. lifetime 应该也没用（生命周期跟着 BulletConfig 走）

## 调查结果

### 第 1 条「线没连」— 经拓扑分析证伪

红队对 v0.3 做了完整静态分析（节点 89 / 边 115）：
- **完全孤立节点 0**
- **不可达节点 0**（双根 BFS 覆盖全部 78 个非 tag 节点）
- **ConfigJson Params ↔ edges 完全一致**（无暗引用）

结论：跟 PostMortem #022 同款症状 — **双根模板设计意图导致 OnTick 子图与 L1 主流程视觉不连**，并非真断链。这次不是 bug。

### 第 2 条「改了参数无效」— 真根因定位

**根因**：v0.3 的 TSCT_VALUE_COMPARE 操作符 op 编码全部错配。

v0.3 builder 第 178-190 行注释明确写"empirically 推断自 30212010/30532001/302001 真实样本"，推断结果：
```
op=1 ↔ "==", op=2 ↔ "!=", op=3 ↔ "<", op=4 ↔ "<=", op=5 ↔ ">", op=6 ↔ ">="
```

但 C# 真值（`Assets/Scripts/TableDR_CS/NotHotfix/Gen/common.nothotfix.cs:7929` `TConditionOperator` 枚举）：
```
TCO_NULL              = 0   "-"
TCO_EQUAL             = 1   "=="
TCO_NOT_EQUAL         = 2   "!="
TCO_BIGGER            = 3   ">"
TCO_BIGGER_OR_EQUAL   = 4   ">="
TCO_SMALLER           = 5   "<"
TCO_SMALLER_OR_EQUAL  = 6   "<="
TCO_OR/AND/NOT/CLOSED_INTERVAL/INCLUDE/HAS_FLAG = 7..12
```

**v0.3 的 < 跟 >、<= 跟 >= 全部反向了！** 用户的 maxRadius 改了之后行为反向（应该 R>maxR 销毁却变成 R<maxR 销毁），minRadius 同款。followPlayer 用 op=2 (!=) 恰好对所以"分支"看着没事，掩盖了根因。

为什么 empirical 推断错了：30212010 line 8106 那条 `[V=eid/PT=2, V=2, V=0/PT=0]` 推 `tag != 0` 是对的（op=2），但 302001 BUFF 感电 line 2363 `[V=eid/PT=2, V=6, V=30]` 当时被推为">=30"（op=6 当 >= 用），实际 op=6 是 `<=` 即 ">=值30"实为"<=30"。本来从一两个样本反推就有 50% 反向风险，红队 v0.3 跳过了"读 C# 枚举源码"这一步直接动手实现，是经典 empirical inference 翻车。

### 第 3、4 条「damagePerHit / lifetime 删掉」— 设计简化

用户给出非常合理的设计意见：
- damagePerHit：调用方传 bulletID 后，伤害本来就是 BulletConfig.AfterBornSkillEffectExecuteInfo 的命中链路决定的。模板再单加一层 EXT_PARAM 是冗余（且需要调用方接 SkillTag 32300714 才生效，更繁琐）。删。
- lifetime：子弹的 BulletConfig.LastTime 自己管。模板算 elapsedFrames 强行 DESTROY 是越权。整个 lifetime 销毁机制全删（连同实体级 SkillTag 32300713）。

## 新规则 / 防线

### 规则 1（流程级）：empirical 推断的枚举值必须用 C# 源码二次验证

**新规则**：任何"用 N 个真实样本反推枚举值"的尝试，**必须**在动手实现前 grep C# 源码找权威定义（common.nothotfix.cs / 节点的 ParamsAutoGenerated.cs）。

**根因**：枚举推断从 1-2 个样本看上去对，到大规模生效后才发现某 op 反向，已经写入产线模板。修正成本远高于先 grep 5 分钟。

**实现方式**：沉淀到 `.claude/skills/skill-design/SKILL.md` GATE-0.5 校对清单 — 加一条"`enum E { ... }` 必须找到 C# 源码定义，不可只 empirical 推断"。

### 规则 2（lint 工具，手动启用）：E021 TemplateParam 槽位无消费

**新规则**：扫所有 IsTemplate=true 节点的 TemplateParams 数组，对每个槽位 i 若模板内无节点用 `ParamType=4/Value=i` 引用 → 报 E021 ERROR。

**根因**：暴露给调用方的 EXT 参数槽位若没人读，调用方面板上能填值但运行时无效，是"沉默 bug"。v0.3 之前的迭代（v0.1→v0.2）都是这个状态——只声明不消费。

**降级原因**：在 118 个真实模板上测试假阳性率 31%（37/118）。生产模板大量有"暴露但当前未消费"的合法用法（预留扩展点 / 跨文件引用 / 历史保留）。强制 ERROR 会污染默认 lint 管线。

**实现**：[skill_lint.py:650-720](../tools/skill_lint.py)（`check_e021_template_param_unused`）。**默认未注册到 `lint_file`**。仅在 builder 自检时手动调用：

```python
from skill_lint import check_e021_template_param_unused, LintReport
report = LintReport(file=path)
check_e021_template_param_unused(refs, report)
if report.errors:
    sys.exit("E021 触发，模板有暴露未消费槽位")
```

**护栏验证**：
- v0.4 真模板默认 lint：E=0（PASS）
- v0.4 真模板手动 E021 检查：0 errors（10/10 全消费 PASS）
- 故障注入（追加 2 个虚假槽位 [11]/[12]）：手动 E021 检查 2 errors（成功捕获）
- 真实样本 118 个递归默认 lint：未引入新假阳性（因为 E021 没在默认管线里）

## 反向反思（最重要的一条）

**接力消息错误诊断的本质**：用户报"线没连"+"改了参数无效"被默认推断为"v0.3 新加节点真的有边漏写"。如果按这个诊断动手，会反复改 builder 但永远看不到效果——因为根因不在拓扑断链而在 op 编码。

**红队应有的反应（已做到）**：
1. 不直接照单全收"修边"的指令
2. 先做完整静态分析（参数槽消费 / 孤立节点 / 双根可达性 / ConfigJson↔edges 一致性 4 项）
3. 拓扑层面 100% 干净后再深入查 v0.3 新加节点的语义层面（CONDITION_EXECUTE Params + VALUE_COMPARE 真值）
4. 发现 op=5 (理论 >) 却用在"R<minRadius" 处的不一致后，反向追到 C# 真值
5. 找到 `Assets/Scripts/TableDR_CS/NotHotfix/Gen/common.nothotfix.cs:7929 TConditionOperator` 权威定义后才动手改

**这条反思要沉淀到 SKILL.md GATE-0.5**：
> 「枚举值校对」必须查 C# 源码权威定义，不能只 empirical 推断 1-2 个样本。
