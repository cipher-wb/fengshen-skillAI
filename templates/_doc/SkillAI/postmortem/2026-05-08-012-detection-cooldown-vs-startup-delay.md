# v1.3 实验失败 — "侦测冷却"不是启动延迟，子弹出生即死

## 决策
✅ **入库 v1.3.1（DELAY 包第二碰撞模板）**

## 用户反馈（v1.3 实测）
**子弹看不到了，是不是出生就死了？**

## 根因 — 错误假设"侦测冷却"语义

### v1.3 设计（错）
```python
collision_self_node = make_template_call_node(
    template_key="子弹通用逻辑-碰撞",
    extra_params=[
        ...,
        flight_frames + hover_frames,   # [11] 侦测冷却 = 70 帧
        ...
    ],
)
```

我以为 `Params[11] 侦测冷却` 是**启动延迟**（等 70 帧后才开始检测），但实际语义是**两次成功侦测之间的冷却时间**。

### 实际行为
1. 子弹生成 → 第二碰撞模板**立即开始检测**
2. 子弹生成位置 ≈ 主角位置（0 距离）
3. target_camp=3（创建者）匹配主角 → 立即触发命中后功能
4. 命中后功能 = `TSET_DESTROY_ENTITY` 销毁主体（子弹自身）
5. 子弹**出生瞬间死亡** → 用户看不到

侦测冷却 70 帧仅在**第二次检测前**生效（第一次检测就死了，根本没第二次）。

## 解决方式（v1.3.1）

用 **TSET_DELAY_EXECUTE** 包裹第二碰撞模板，飞出+悬停阶段不挂载：

```
AfterBorn = ORDER:
  ├ 碰撞模板1（敌方）— 立即生效
  └ TSET_DELAY_EXECUTE (frames = flight + hover + 5):
        └ 碰撞模板2（target_camp=3 主角）— 75 帧后才挂载
```

子弹出生时**第二碰撞模板还没有挂上**，飞出+悬停阶段安全。回飞前 5 帧才挂主角检测。

## 编译产物（30142002 v1.3.1）
- 节点数：31（v1.3 是 30）
- 多 1 个 TSET_DELAY_EXECUTE 节点

## 利
- 子弹不会出生即死
- 飞出+悬停阶段没有主角碰撞干扰
- 回飞阶段开始检测主角，撞到才销毁
- "分别消失" 的精确语义有望实现（每颗子弹各自的 DELAY 启动）

## 弊
- target_camp=3 是否真能精确检测**仅创建者**仍未实测验证
- 如果 target_camp=3 实际行为与预期不符（如把所有人都当"自己"），子弹会被错误销毁

## 噪音风险
中 — DELAY + target_camp=3 组合是创新拓扑，需要实测确认

## 反思 — 字段语义"望文生义"的代价

| 字段名 | 我以为 | 实际是 |
|---|---|---|
| **侦测冷却** | 启动延迟（等 N 帧后开始） | 两次成功检测间的冷却 |
| **TTriggerType** | TShapeType（1=圆）| 0=圆形 |
| **FlyType=6 追踪坐标点** | 自动追施法者 | 朝面向飞 |

3 次踩坑模式相同：**字段名简洁但含义需要查代码**。

**沉淀更新**：[skill-design SKILL.md GATE-0.5 §3](../../../.claude/skills/skill-design/SKILL.md) 加新警告：
> 模板/BulletConfig 字段名只是 hint，不能替代真实行为验证。任何带"冷却/延迟/启动"等模糊语义的字段，**必须找 ≥1 个真实样本验证默认行为**，不能凭直觉猜。

## Unity 实测预期（v1.3.1）
1. 子弹**正常飞出**（不再出生即死）
2. 飞出 30 帧 + 悬停 40 帧期间，**主角靠近也不会销毁子弹**（DELAY 未到，第二碰撞模板未挂）
3. 75 帧后回飞开始 → 子弹靠近主角时**逐颗销毁**（路径微差异让陆续触发）

如果 #3 仍同步消失 → target_camp=3 不能区分 8 颗子弹各自的"创建者"，所有子弹同时检测同一主角 → 同时销毁。这种情况 fallback 到 v1.2.4 接受同步消失。
