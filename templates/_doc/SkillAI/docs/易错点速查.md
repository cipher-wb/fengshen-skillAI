## 1. TableTash 用错 — 节点静默失效

**症状**：节点显示存在，Lint 通过，**但运行时节点逻辑像没生效**（用默认值）。最隐蔽的 bug，因为没有任何错误提示。

**根因**：SkillEditor 每张表有独立的 TableTash（schema hash）。如果给 SkillConditionConfig 节点（如 TSCT_VALUE_COMPARE）用了 SkillEffectConfig 的 TableTash，编辑器反序列化时 hash 不匹配 → Config 反序列化为 null → 节点 GetValue() 返回默认值 → 行为悄悄错。

**已知 TableTash 速查**：

| 表 | TableTash |
|----|-----------|
| SkillEffectConfig | `0CFA05568A66FEA1DF3BA6FE40DB7080` |
| SkillTagsConfig | `6A8A6883BDFDA1411BB2461E65CB2D9B` |
| **SkillConditionConfig** | `ED89F46EAB95F7ACF5C1911A5A375278` |
| ModelConfig | `B1156BD6A276DA7A8E082421FCF0B7A9` |
| BulletConfig | `A4AEA1F4B1BD0FA9FD2F066BE902466F` |

**额外要点**：
- TSCT_VALUE_COMPARE 等条件节点必须有顶层 `SkillConditionType` 字段（**不是** `SkillEffectType`）
- Config2ID 前缀也必须是对应表名：`SkillConditionConfig_<id>` 不是 `SkillEffectConfig_<id>`

**修法**：编辑器手配时**不会出现这个问题**（编辑器自动选对 TableTash）。只在**用 Python 脚本生成 JSON** 时容易踩坑。

## 2. PT=5 是 TCommonParamType（不是"槽位"）

**症状**：写"伤害归属用 4"和"用 6"行为完全不同，但都不报错。

**根因**：Param 的 `ParamType=5` 表示 `TCommonParamType`（运行时上下文枚举），常用值：

| Value | 含义 |
|-------|------|
| 1 | 主体单位实例 ID（TCPT_MAIN_ENTITY） |
| 2 | 目标单位实例 ID（TCPT_TARGET_ENTITY） |
| 3 | 施法者实例 ID（TCPT_CREATE_ENTITY） |
| 4 | 主体单位-伤害属性归属单位 |
| 5 | 目标单位-伤害属性归属单位 |
| 6 | 施法者-伤害属性归属单位 |
| 7 | 初始技能 ID |

**绝对不是 "槽位编号"**。早期常有人误解为"第 X 个目标"，错得离谱。

**修法**：
- 子弹的"伤害来源"应该是 V=6（施法者伤害归属，因为子弹脱离施法者飞行）
- IR 写法：`entity:施法者伤害归属` 或简写 `entity:6`

## 3. 子弹必须走标准命中链路

**症状**：子弹打到目标了，但目标没掉血/没有命中表现/没飞溅特效。

**根因**：MOBA 子弹的命中伤害**不能**用 `TSET_DAMAGE_QUICK` 直接造，必须走"碰撞模板 → 命中后 ORDER → 伤害模板 + 表现模板"的标准链路。

**完整拓扑**：
```
TSET_CREATE_BULLET
    └ Params[0] BulletID → BulletConfigNode
                            └ AfterBornSkillEffectExecuteInfo.SkillEffectConfigID
                                ↓
                            子弹通用碰撞模板 (TSET_RUN_SKILL_EFFECT_TEMPLATE)
                              路径: 技能模板/子弹/SkillGraph_【模板】子弹通用逻辑-碰撞.json
                              模板根 ID = 190016404
                              Params[3] 命中后功能 → ORDER
                                                      ├ 伤害模板（路径: 子弹通用逻辑-伤害.json, 根=190016485）
                                                      └ 表现模板（路径: 子弹通用逻辑-表现.json, 根=190016523）
```

**修法**：用 [skill-design](../../../.claude/skills/skill-design/SKILL.md) Skill 生成时会自动加链路。手配时参考 [docs/模板库.md](模板库.md) 的"直线子弹"原型。

## 4. BulletConfig / ModelConfig 必须显式拉节点

**症状**：技能创建子弹后，编辑器节点上**只显示一个 ID 数字**，看不到子弹的速度、模型、生命周期等属性。

**根因**：TSET_CREATE_BULLET 的 BulletID 字段、TSET_CREATE_EFFECT 的 ModelID 字段，在 SkillEditor 中**应该连到一个独立的 BulletConfigNode / ModelConfigNode 节点**，而不是只填一个 ID 数字。

**修法**：在编辑器里拉一个 BulletConfigNode 出来，把 ID 节点连到 CREATE_BULLET 的 BulletID 字段。Python 生成时用 `make_bullet_config_node` 自动生成。

**附属陷阱**：BulletConfigNode 的 ConfigJson **必须完整**（含 BeforeBornSkillEffectExecuteInfo / AfterBornSkillEffectExecuteInfo / DieSkillEffectExecuteInfo 三个 dict）。Python 脚本生成时优先**从已有 SkillGraph JSON 拷贝**，降级才查 xlsx — 编辑器保存过的数据 100% 合法。

---

## 5. SkillConfig 必须填 MOBA 通用默认值

**症状**：技能在游戏里 AI 不会用、没法选目标、释放时没有指示器。

**根因**：SkillConfigNode 的下列字段不能留空：
- `SkillIndicatorType` + `SkillIndicatorParam`：施法指示器
- `SmartCastTarget*`（4 个）：智能施法（AI 用）
- `CastTargetCondTemplate` + `CastTargetCampCond` + `CastTargetMonsterRankCond`：玩家释放目标限制

**默认值（从真实样本提炼）**：
- 指示器类型：直线子弹→2 / 圆形 AOE→3 / 否则→0
- 指示器参数：默认 `[skill.range]`
- 智能施法优先级：3（距离最近）
- 目标条件模板：1
- 怪物等级条件：`[16, 15, 14, 13, 12, 11, 4, 5, 6, 0]`
- 阵营条件：`[2]`（敌方）

## 6. TSET_ORDER_EXECUTE 的边 outputPortIdentifier 必须 "0"

**症状**：节点视觉上"悬空"——子节点和 ORDER 没有连线，但运行时**功能正常**。

**根因**：xNode/GraphProcessor 框架中，`dynamicPortList` 风格的节点（端口数量可变长，如 `TSET_ORDER_EXECUTE`、`TSET_NUM_MAX`、`TSET_NUM_MIN`）的所有出边 `outputPortIdentifier` 必须**全部为字符串 "0"**，而不是 `Params[N]` 的索引。

**修法**：手配时编辑器自动处理。Python 生成时维护 `DYNAMIC_PORT_NODES` 集合，强制设为 `"0"`。

## 7. SkillEditor 模板不需要 SkillConfigNode 外壳

**规则**：制作 SkillEditor 模板（`Saves/Jsons/技能模板/`）时，**根节点应该直接是 ORDER (IsTemplate=true)**，不需要 SkillConfigNode 外壳。

**原因**：模板的语义是"被其他技能图通过 RUN_SKILL_EFFECT_TEMPLATE 调用的可复用片段"，不是独立技能。SkillConfigNode 用于普通技能的 SkillConfig 表行注册——模板没有 SkillConfig 对应行。

**注意**：旧模板（如 1860196 多向子弹）有 SkillConfigNode 是历史遗留，**新模板不带**。

## 8. 子弹伤害来源应该是 V=6（施法者伤害归属）

**规则**：`子弹通用逻辑-伤害` 模板的 Params[4]「伤害来源」应该是 `entity:施法者伤害归属`（V=6, PT=5），不是 `entity:主体伤害归属`（V=4）。

**原因**：MOBA 子弹脱离施法者飞行，伤害归属要追溯到施法者本人，而非"主体"。

参考真实样本 30222005 验证。

## 9. 模板内孤立分支会报"表格缺失"

**症状**：模板加载后某些 ORDER 节点报错"表格缺失 SkillEffectConfig:XXXXX"，对应 ID 节点其实存在但孤立。

**根因**：编辑器的"表格缺失"校验对所有节点的 Params 引用扫描，但只追溯"被根节点连通的子图"。**孤立分支**（自己有出边但没入边）的节点会被认为"未使用"，进而它们被引用的 ID 也认为"不存在于活跃图中"。

**修法**：编译/生成模板时必须保证**每个非根节点都至少有 1 条入边**。简化版模板时**跳过未实现分支的同时，要删除该分支创建的所有节点**——不能只跳过引用，节点本身也要删。

## 10. SkillEditor 模板根 EID 要固定（避免列表里出现重复）

**症状**：每次重新生成模板，编辑器工具栏的"模板列表"里就多一份同名模板（不同 ID）。

**根因**：编辑器 [JsonGraphManager.cs:1268](../../../Assets/Thirds/NodeEditor/Datas/JsonGraphManager.cs#L1268) 的 `RemoveNodeInfo` 漏了清 `template2Nodes` 字典。如果生成器每次给模板根分配的 EID 不同（自动递增），老 EID 永远不被删，新 EID 不断累加。

**修法**：模板生成器给根节点用**固定 EID**（不要自动递增到末尾），这样后续重生成只覆盖同一个 key，不再积累。

清掉残留：重启 Unity Editor（in-memory 字典从磁盘重建）或删 `NodeEditor/Cache/JsonGraphManager_v*.json`。

## 11. 子弹碰撞模板的 collision_radius 已是 AOE，不要画蛇添足造 QUICK_DAMAGE

**症状**：自己加 `TSET_QUICK_DAMAGE` 想"实现 AOE 范围伤害"，结果一次命中造**双份伤害**（直伤模板 + QUICK_DAMAGE 各一份）。

**根因**：子弹通用-碰撞模板（190016404）的 `collision_radius` 已经是 **AOE 检测半径**——碰撞模板会**对范围内每一个目标各调用一次伤害模板**。这本身就是天然 AOE 多目标伤害。

**正确机制**（一图读懂）：

```
子弹通用-碰撞模板 collision_radius=200
  ├ 检测圆内目标 A → 调用伤害模板（A 受伤）
  ├ 检测圆内目标 B → 调用伤害模板（B 受伤）
  └ 检测圆内目标 C → 调用伤害模板（C 受伤）
   总共造 3 份伤害（每个目标 1 份），不需要额外节点
```

**对照表**：

| 错误做法 | 正确做法 |
|---|---|
| 配 hit.damage（直伤）+ 加 QUICK_DAMAGE 节点（"AOE 量"）| 只配 hit.damage（碰撞半径决定范围 + 伤害模板天然多目标） |
| 一个目标受双份伤害 | 一个目标受一份伤害；范围内每个目标各受一份 |
| 数值评估按"直伤+爆炸"双叠加 | 数值评估按"伤害模板单次伤害" |

**IR 写法**（v1.1 修正版 — explosion 字段是语义糖）：

```yaml
hit:
  explosion:
    radius: 200            # → 映射成 collision_radius=200（AOE 检测半径）
    damage:                # → 覆盖直伤模板的 coef/element/type 等
      coef: 8000
      element: 火
      type: 法术
  on_hit_buff:             # 命中目标全部挂 buff
    buff_id: 1080449
    duration_frames: 150
```

**编译产物**：collision_radius=200 + 伤害模板 coef=8000 + 挂 buff 节点。**不再有 QUICK_DAMAGE 节点**。

**已纳入**：[skill_compiler.py](../tools/skill_compiler.py) v1.1.1 修正：
- explosion.radius → collision_radius（AOE 检测半径）
- explosion.damage → 覆盖伤害模板参数
- 不再生成额外 QUICK_DAMAGE 节点

## 12. SkillIndicatorParam 按 SkillIndicatorType 分类填值

**症状**：单目标指示器（type=1）填了 `[range]`，编辑器/运行时报指示器异常或显示错误。

**根因**：不同指示器类型期望的参数不同：

| SkillIndicatorType | 含义 | 正确 SkillIndicatorParam |
|---|---|---|
| 0 | 无目标 | `[]` |
| 1 | **单体（单目标）** | `[]`（目标已锁定，无范围参数） |
| 2 | 直线 | `[range]` |
| 3 | 圆形 | `[range]` |
| 4 | 扇形 | `[range, angle]` |
| 5 | 矩形 | `[width, height]` |

**修法**：手配时按表填；用 IR 时编译器自动按 type 分类（v1.1.1 已修正）。

**已纳入**：[skill_compiler.py make_skill_config_node](../tools/skill_compiler.py)：indicator_param 现按 type 分类生成。

## 13. 技能时长规则（4 条全约束）

**症状**：编辑器报错 `[ID:XXXXX] 技能时间不符合规则`。

**根因**：SkillEditor 内部时长校验有 **4 条约束**：

| # | 约束 | 含义 |
|---|------|------|
| 1 | `cast ≤ base_duration` | 出手帧不能超出基础时长（不然动画结束了还没出手） |
| 2 | `cast ≤ buffer_start + buffer_frame`（buffer_frame > 0 时） | **出手必须在缓冲区结束前**（出手后还能在缓冲窗口被取消/连招） |
| 3 | `buffer_start ≤ base_duration`（buffer_start > 0 时） | 缓冲起始也不能超出基础时长 |
| 4 | `base_duration ≤ cd_time` | 技能本身不能比 CD 还长 |

**典型违规**：
- cast=24, bd=20 → 违反 #1（24 > 20）
- cast=24, bs=0, bf=20 → 违反 #2（buffer_end=20 < cast=24）⚠️ **最容易漏的一条**
- bd=90, cd=60 → 违反 #4（90 > 60）

**字段映射**：

| IR 字段 | SkillConfig.json 字段 | 中文 |
|---|---|---|
| `cast_frame` | SkillCastFrame | 出手帧 / 攻击触发帧 |
| `buffer_start_frame` | SkillBufferStartFrame | 缓冲区起始帧 |
| `buffer_frame` | SkillBufferFrame | 缓冲帧（窗口长度） |
| `base_duration` | SkillBaseDuration | 基础时长 |
| `cd_frames` | CdTime | CD |

**修法**：
- 优先调大 `buffer_frame` 让 `buffer_start + buffer_frame >= cast`
- 调高 `base_duration` 让 `cast <= bd`
- 调高 `cd_frames` 让 `bd <= cd`

**真实样本参考（火宗门）**：
- 304006: cast=16/bs=0/bf=20/bd=20/cd=40 — cast(16) ≤ buffer_end(20) ✓
- 30142001: cast=24/bs=0/bf=30/bd=30/cd=90 — cast(24) ≤ buffer_end(30) ✓

**已纳入**：
- [skill_lint.py E013](../tools/skill_lint.py)：4 条约束全部检查（2026-05-08-004 补加 #2）
- 速查表本条

## 14. 回旋飞回 pattern 必须双 Bullet 接力实现

**症状**：用户想做"飞出阶段无伤害 / 回飞阶段才有伤害"（LOL 泰隆 R 风格），单 BulletConfig 配不出来。

**根因**：BulletConfig 只有 3 个生命周期钩子：
- BeforeBornSkillEffectExecuteInfo（生成前）
- AfterBornSkillEffectExecuteInfo（生成后整个生命周期）
- DieSkillEffectExecuteInfo（死亡时）

**没有"折返时" / "速度反向时"等中间钩子**。所以单 BulletConfig 的 AfterBorn 要么全程挂碰撞要么全程不挂，无法"飞出不挂 / 回飞才挂"。

**真实样本验证**：
- 30212005 木影旋刃 / 22017101 蝴蝶妖回旋镖 都是"碰到造伤害+触发回飞"模式（碰撞与回飞**同时**），**不是**真正的"飞出无伤 / 回飞才有伤"

**解决方案**：双 Bullet 接力（v1.2 IR 已封装为 `pattern: 回旋飞回`）

```
Bullet1（飞出+悬停）：
  FlyType=5/TracePathType=4 (静态轨迹)
  AfterBorn = 空 ← 飞出阶段不挂任何效果（无伤害）
  Die = ORDER → 在死亡位置创建 Bullet2

Bullet2（回飞）：
  FlyType=6 (TRACK_TARGET_POINT) 追施法者位置
  AfterBorn = 子弹通用-碰撞模板 ← 回飞阶段才挂碰撞（有伤害）
```

**IR v1.2 用法**（用户视角）：

```yaml
- bullet:
    pattern: 回旋飞回                     # 触发双 Bullet 接力
    angles: [0, 45, 90, 135, 180, 225, 270, 315]   # 一圈 8 颗
    boomerang:
      flight_frames: 30                   # 飞出帧
      hover_frames: 20                    # 悬停帧
      max_distance: 600                   # 最远距离
      return_speed: 2000                  # 回飞速度
    hit:
      collision_radius: 100
      damage: { coef: 12000, type: 法术 }
```

编译器自动生成 Bullet1 / Bullet2 / Die ORDER / 命中链路全套。

**已纳入**：

## 15. 技能威力系数 + 额外伤害必须成对（铁律）

**症状**：技能首节点 `SkillParamList` 里只配了"技能威力系数"（倍率伤害），漏配"技能额外伤害"（固定值伤害）。运行时不报错，但伤害结构不完整 → 实际伤害低于策划预期。

**根因**（用户原话定为铁律）：
1. 项目所有技能（含近战）的伤害都通过"子弹"造成
2. 一次伤害 = **倍率部分（技能威力系数）+ 固定部分（技能额外伤害）**，缺一不可
3. 多段伤害用不同子弹序号区分（1/2/3 号常用，最大 5 号）
4. 普通伤害走通用模板 `SkillGraph_【模板】子弹通用逻辑-伤害.json`，模板内"本子弹序号"决定首节点要配哪一对 Tag

**Tag ID 配对表**（出现左列必须出现右列）：

| 子弹序号 | 技能威力系数 ID | 技能额外伤害 ID |
|---------|----------------|----------------|
| 1 号 | **3** | **5** |
| 2 号 | **1201** | **1211** |
| 3 号 | **1202** | **1212** |

**审核检查**：每个技能 SkillConfigNode 的 `SkillParamList` 必扫——
- ID=3 必须配 ID=5
- ID=1201 必须配 ID=1211
- ID=1202 必须配 ID=1212
- 子弹模板用了"本子弹序号 N" → 首节点必须配第 N 号子弹的成对参数

**例外**：少数特殊逻辑技能不走通用伤害模板，可豁免；豁免要在 PostMortem 标注理由。

**修法**：缺哪一半补哪一半，值由策划决定（一般固定值 1000~3000 区间起步）。

**真实样本**：30212009 千叶散华（2026-05-09 修复）—— 原本只有 ID=3、ID=1201，补齐 ID=5、ID=1211 两条。
