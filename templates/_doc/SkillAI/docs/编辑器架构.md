# 技能编辑器架构

## 概述

`SkillEditor` 是修仙时代项目的技能蓝图系统，基于 [[xNode节点编辑器]] 同源的 GraphProcessor 框架实现。它的核心定位是**编辑期可视化、运行期表格驱动**：

- **编辑期**：策划用图形化节点连线配置技能，节点之间通过 GUID + Params 数组传递参数
- **存盘**：每个技能保存为一份独立 JSON（`SkillGraph_<ID>....json`），节点数据存到 `references.RefIds[]` 数组
- **运行期**：所有节点的 `ConfigJson` 字段会反序列化为 `SkillEffectConfig` / `SkillSelectConfig` / `SkillConditionConfig` 等表行（在 `TableDR_CS` 中），战斗逻辑按 `SkillEffectType` 等 int 枚举 switch 分发到处理函数。**TSET_* 等 partial class 仅在编辑期存在**，运行期不创建实例。

也就是说：技能蓝图编辑器是个"配表工具的图形外壳"，最终产物是结构化的表格数据。

## 核心概念

### 三阶段流水线

```
图形配置（编辑期）        →     表格行（存档）        →     战斗运行（运行期）
─────────────────              ──────────────              ──────────────
SkillGraph (xNode)                SkillConfig表              BattleSkillProcessor
TSET_*/TSCT_*/TSKILLSELECT_*     SkillEffectConfig表          按 SkillEffectType
.Custom.cs (UI/校验)              SkillSelectConfig表          int 枚举 switch
                                  SkillConditionConfig表       到对应处理函数
                                  SkillTagsConfig表
                                  ConfigJson (字符串内嵌)
```

### 5 大主节点（BaseConfig）

技能图的"骨架"节点，每种对应一张表：

| 主节点类 | 对应表 | 职责 | 一图至多几个 |
|---------|-------|------|-------------|
| `SkillConfigNode` | `SkillConfig` | **技能根**：ID/CD/范围/前摇/缓冲区/施法条件/连招/AI 智能施法 | 1 |
| `SkillTagsConfigNode` | `SkillTagsConfig` | **技能参数定义**：参数 schema（key=ID, value=int），支持死亡保留/最终值计算 | 多 |
| `SkillSelectConfigNode` | `SkillSelectConfig` | **筛选根**：按形状/阵营/数量找战斗单位 | 多 |
| `SkillEffectConfigNode` | `SkillEffectConfig` | **效果根**：所有 TSET_* 都继承它 | 多 |
| `SkillConditionConfigNode` | `SkillConditionConfig` | **条件根**：所有 TSCT_* 都继承它 | 多 |
| `SkillInterruptConfigNode` | `SkillInterruptConfig` | **中断配置根**：用于延迟/重复执行的中断信号包 | 多 |

### 子节点三大门类

主节点是基类，**实际配置时拉的是子节点**：

| 子节点前缀 | 含义 | 数量级 | 目录 |
|----------|------|--------|------|
| `TSET_*` | T(ype) Skill Effect Type — 效果节点 | ~250 种类型（编辑期 26 个定制类） | `Nodes/SkillEffectConfig/` |
| `TSKILLSELECT_*` | 筛选节点 | ~22 种类型（编辑期 2 个定制类） | `Nodes/SkillSelectConfig/` |
| `TSCT_*` | T(ype) Skill Condition Type — 条件节点 | ~95 种类型（编辑期 4 个定制类） | `Nodes/SkillConditionConfig/` |

> 编辑期定制类数量远少于运行期类型数量——**多数节点没有 .Custom.cs**，UI 端口靠反射 + `TableAnnotation.json` 注解动态生成。定制类只是为了加保存校验、动态参数、模板支持等高级特性。

### 引用节点 RefConfigBaseNode

**不是真节点**，是"我引用别处定义的表格行"的纯指针。常见用法：在 SWITCH/ORDER 等流程节点中需要引用别处的 `SkillEffectConfig` ID 但不复制内容。

字段：`TableManagerName`（如 `TableDR.SkillEffectConfigManager`）+ `ID`（手动填）+ `ManualID`。

`JsonGraphManager` 把它放进单独的 `RefConfig2Nodes` 索引，不与正常节点混淆。

## JSON Schema

### 顶层结构

```jsonc
{
  "serializationData": { ... },                 // Odin 二进制容器，技能图基本不用
  "nodes": [ {"rid": 1000}, {"rid": 1001}, ... ], // 节点 rid 引用列表
  "edges": [ ... ],                              // 节点连线（详见下）
  "groups": [ ... ],                             // 视觉分组框（无逻辑）
  "stackNodes": [], "stickyNotes": [],
  "pinnedElements": [], "exposedParameters": [],
  "serializedParameterList": [],
  "curTab": 0,
  "path": "Assets/.../SkillGraph_xxx.json",
  "references": {
    "version": 2,                               // Unity 2021+ RefIds 格式
    "RefIds": [
      { "rid": 1000, "type": {...}, "data": {...} },
      ...
    ]
  }
}
```

### 节点（references.RefIds[i]）

```jsonc
{
  "rid": 1000,                          // 自增整数，仅本图内唯一
  "type": {
    "class": "SkillConfigNode",         // 节点类名，反射定位 C# 类
    "ns": "NodeEditor",
    "asm": "NodeEditor"
  },
  "data": {
    "GUID": "<uuid>",                   // 跨文件全局唯一身份证
    "computeOrder": 0,                  // 编辑期连线传播顺序
    "position": { "x": ..., "y": ... },
    "expanded": true,
    "ID": 30122001,                     // 表格 ID（这就是技能 ID）
    "Desc": "技能描述",
    "IsTemplate": false,                // 是否为模板节点
    "TemplateFlags": 0,
    "TemplateParams": [],
    "TableTash": <uint32>,              // 表结构 hash，结构变更时检测
    "ConfigJson": "<字符串化的子 JSON>",  // 关键：表格行的全部字段
    "Config2ID": "SkillConfig_30122001", // 全局索引 key
    // 类型特有字段：
    "SkillEffectType": <int>,           // 仅 SkillEffectConfigNode/TSET_*
    "SkillConditionType": <int>,        // 仅 SkillConditionConfigNode/TSCT_*
    "SkillSelectType": <int>,           // 仅 SkillSelectConfigNode/TSKILLSELECT_*
    // ITemplateReceiverNode 节点额外：
    "TemplateData": {
      "TemplateParams": [...],
      "TemplatePath": "Assets/.../技能模板/.../SkillGraph_xxx.json"
    }
  }
}
```

### ConfigJson（关键字段）

所有节点的"业务字段"都序列化在 `ConfigJson` 字符串里，**延迟反序列化**（`NodeInfo.Config` 是 lazy property）。结构因表而异：

- `SkillConfig` 的 ConfigJson：`{ID, SkillNameKey, ElementType, SkillMainType, SkillSubType, CdType, CdTime, SkillRange, SkillEffectExecuteInfo, SkillTagsList, ...}`，非常宽
- `SkillEffectConfig` 的 ConfigJson：`{ID, SkillEffectType, Params:[{Value,ParamType,Factor},...]}`，定长字段 + 变长 Params
- `SkillTagsConfig` 的 ConfigJson：`{ID, Desc, TagType, NameKey, DefaultValue, FinalValueEffectID, RetainWhenDie}`

### 连线（edges）

连线本质是"父节点 Params[X] 槽位 → 子节点 ID"，不是图形化的"流程箭头"：

```jsonc
{
  "GUID": "<edge-uuid>",
  "inputNodeGUID":  "<目标节点 GUID>",      // 子节点
  "outputNodeGUID": "<源节点 GUID>",        // 父节点
  "inputFieldName": "ID",                  // 子节点的 ID 字段
  "outputFieldName": "PackedParamsOutput", // 父节点的 Params 输出端口
  "inputPortIdentifier": "0",
  "outputPortIdentifier": "5",             // 父节点 Params[5] 的位置
  "isVisible": true
}
```

两种 outputFieldName：

- `PackedParamsOutput` + `outputPortIdentifier: "<index>"` = 父节点 `Params[index]` 输出。常见于 TSET_*/TSCT_*/TSKILLSELECT_* 之间。
- `PackedMembersOutput` + `outputPortIdentifier: "<member-path>"` = 父节点的命名字段（如 `SkillEffectExecuteInfo.SkillEffectConfigID`）输出。常见于 SkillConfigNode 挂载入口效果。

### TParam 三元组

`SkillEffectConfig.Params` 等数组的每一项是 `TParam = {Value:int, ParamType:enum, Factor:int}`：

| 字段 | 用途 |
|------|------|
| `Value` | 主载荷：数值、表格 ID、引用 ID 等 |
| `ParamType` | 类型鉴别枚举（`TableDR.TParamType`），决定 Value 的语义 |
| `Factor` | 加成系数 / 附加 ID（如某些参数把 `Factor` 当作 `SkillTagsConfig` 引用 ID） |

**ParamType 完整定义**（来自 `Assets/Scripts/TableDR_CS/NotHotfix/Gen/common.nothotfix.cs`）：

| 整数 | 枚举名 | 中文 | Value 引用 |
|------|--------|------|-----------|
| 0 | `TPT_NULL` | - | 纯数值或表格 ID（最常见） |
| 1 | `TPT_ATTR` | 属性 | → `TBattleNatureEnum`（运行时读单位属性表，例如 V=59 = 位置X、V=60 = 位置Y、V=91 = 面向） |
| 2 | `TPT_FUNCTION_RETURN` | 函数返回值 | → 另一份 `SkillEffectConfig` 的返回值（嵌套求值） |
| 3 | `TPT_SKILL_PARAM` | 技能参数 | → `SkillTagsConfig` ID |
| 4 | `TPT_EXTRA_PARAM` | 模板参数 | → 模板宿主传入的参数索引 |
| 5 | `TPT_COMMON_PARAM` | 常用数值参数（**上下文常量**） | → `TCommonParamType`（运行时上下文，常见 V=1 主体单位实例ID/2 目标单位实例ID/3 施法者实例ID/4 主体伤害归属单位...） |
| 6 | `TPT_EVENT_PARAM` | 技能消息参数 | → 当前事件回调的参数槽 |
| 7 | `TPT_COMMON_SKILL_PARAM` | 技能参数-常用 | → 通用技能参数 |

> **注意：之前误把 PT=5 解释为"槽位"——更准确的描述是 `TCommonParamType` 枚举（46 项），见 `common.hotfix.cs` line 3710+`。例如 V=1 = `TCPT_MAIN_ENTITY` 主体单位实例ID。**

## 节点之间怎么挂载（拓扑）

### SkillConfigNode 的入口字段

技能根节点有 6 个主要"入口"，挂载到不同生命周期阶段的效果：

```
SkillConfigNode (根)
├─ SkillEffectExecuteInfo
│    ├ SelectConfigID         → SkillSelectConfigNode（释放时筛选目标）
│    └ SkillEffectConfigID    → 任意 TSET_*（主动释放执行）
├─ SkillEffectPassiveExecuteInfo
│    └ SkillEffectConfigID    → 任意 TSET_*（被动/装备时执行）
├─ SkillEffectOnUnEquip       → 任意 TSET_*（卸下时执行）
├─ SkillEffectOnSkillCastInterrupt → 任意 TSET_*（被打断时执行）
├─ Condition                  → TSCT_*（释放条件）
├─ AICastCondition            → TSCT_*（AI 自动施法条件）
├─ SkillTagsList[].SkillTagConfigID → SkillTagsConfigNode（参数声明）
├─ SkillRangeTagConfigId      → SkillTagsConfigNode（动态施法距离）
├─ SmartCastTargetCondTemplate → TSCT_*（AI 智能选目标条件）
└─ ComboCdList[]              → 内嵌结构体（连招每段独立 CD）
```

### TSET_* 内部的 Params 数组

效果节点用 `Params: List<TParam>` 挂子节点。`Params` 数组的每一项可以是：
- 普通数值
- 引用别的 `SkillEffectConfig` ID（→ 形成"流程嵌套"）
- 引用 `SkillTagsConfig` ID（参数读写）
- 引用 `SkillSelectConfig` ID（筛选）
- 引用 `SkillConditionConfig` ID（条件分支）

### 流程控制的 4 个核心节点

蓝图的"控制流"靠以下节点拼成：

| 节点 | 作用 | Params 含义 |
|------|------|-------------|
| `TSET_ORDER_EXECUTE` | 顺序同步执行多个子效果 | 全部为 SkillEffectConfigID |
| `TSET_DELAY_EXECUTE` | 延迟若干帧执行 | [延迟帧, 子效果ID, 死亡继续, 筛选ID, 中断ID, 急速影响] |
| `TSET_REPEAT_EXECUTE` | 间隔重复执行 | [间隔帧, 次数, 立刻执行, 子效果ID, 筛选ID, 中断ID, 死亡继续, 结束效果ID, 战斗结束继续, 急速影响] |
| `TSET_SWITCH_EXECUTE` | 按值匹配 case | [Switch值, Default效果, (Case1值, Case1效果), (Case2值, Case2效果), ...] 最多 10 组 |

加上**模板节点**：

| 节点 | 作用 |
|------|------|
| `TSET_RUN_SKILL_EFFECT_TEMPLATE` | 跳到外部模板执行（关键复用机制） |
| `TSET_PROBABILITY_EXECUTE` | 概率分支 |
| `TSET_CONDITION_EXECUTE` | 条件分支 |

## 求值模型

### 编辑期：xNode 风格的拉流 + 拓扑排序

GraphProcessor 框架本身是"按 `computeOrder` 拓扑排序的拉流图"。`computeOrder` 数字越小越先求值。但**这一切只在编辑期生效**，目的是：

1. 编辑器节点之间联动（A 节点改了，B 节点 UI 跟着刷新）
2. 端口校验、连线校验
3. 序列化时按依赖顺序写出

### 运行期：表格驱动 + int 枚举 switch

**运行期不再使用图结构**。`BattleSkillEffectProcessor` 类（在 `Assets/Scripts/GameApp/HotFix/Game/Battle/`，待详查）拿到 `SkillEffectConfig` 表行，按 `SkillEffectType`（int 枚举）switch 分发到对应处理函数：

```
TSET_ORDER_EXECUTE      → 遍历 Params，对每个 ID 递归调用 ExecuteEffect()
TSET_DELAY_EXECUTE      → 注册 timer，到点调用 ExecuteEffect(Params[1])
TSET_CREATE_BULLET      → 实例化子弹单位
TSET_APPLY_ENTITY_EFFECT→ 给目标加视觉效果
TSET_DAMAGE_QUICK       → 调用伤害结算流程
...（约 250 种枚举值）
```

控制流由节点类型本身实现，不再走"端口求值"那一套。

> **从策划视角**：连线的本质就是"我把 ID 号填进父节点的某个 Params 槽位"。在编辑器里画线，等同于在父节点的 Params 列表里手动填子节点的 ID——**只是图形化更直观**。

## 模板引用机制

### 模板的本质

模板是另一份 SkillGraph.json，存放在 `Saves/Jsons/技能模板/<分类>/`。模板内**至少有一个根节点的 `IsTemplate=true`**，并通过 `TemplateFlags` 标记位决定子模板属性。

`JsonGraphManager` 启动时扫描所有 JSON，把 `IsTemplate=true` 的节点放进 `template2Nodes[ConfigName][ID]`：
- `ConfigName` ∈ {`SkillEffectConfig`, `SkillConditionConfig`, `SkillSelectConfig`}
- `ID` 是模板表格 ID

### 三种模板宿主节点

实现 `ITemplateReceiverNode` 接口的节点：

1. `TSET_RUN_SKILL_EFFECT_TEMPLATE` —— 引用 SkillEffectConfig 模板
2. `TSKILLSELECT_CUSTOM` —— 引用 SkillSelectConfig 模板
3. `TSCT_GET_TEMPLATE_COND_RESULT` —— 引用 SkillConditionConfig 模板

宿主节点的 `TemplateData` 字段：
- `TemplatePath`：模板 JSON 的相对路径
- `TemplateParams[]`：参数声明数组（每个 `{Name, RefTypeName, isConfigId, isEnum, ...}`）
- 实际参数值仍然填在 `Config.Params` 中，从 `TemplateBeginIndex` 起算

### 模板分类（`Saves/Jsons/技能模板/`）

| 分类目录 | 用途 | 典型模板 |
|---------|------|---------|
| `伤害/` | 伤害结算流程 | 通用伤害流程 / 是否暴击检查 / 是否闪避检查 / 化解检查 / 物理吸血 |
| `功能/` | 通用功能模块 | 飞行状态 / 御剑飞行 / 召唤怪物 / 技能品质适配 / 等级配置 |
| `单位/` | 召唤单位 | 召唤分身（通用）/ 跟随战斗单位销毁 |
| `子弹/` | 子弹生成 | 环形扩散子弹 / 多向子弹 / 链子弹 / 圆形/方形/扇形攻击盒 |
| `技能/` | 整段技能逻辑 | 持续朝向目标 / 平滑转向 / 通用绕身盾 / 阻挡墙包围圈 / 技能连招(3段) |
| `数值/` | 数值取值 | 获取护盾值_命力万分比 |

### ID 编码（观察规律）

技能模板的文件名 ID 有两类：
- 长 ID（8 位以上，如 `146002819`）：大类号 + 顺序号
  - `146` 系列：伤害流程相关
  - `186` 系列：功能模块（连携、召唤等）
  - `175` 系列：通用流程
  - `129/136/66` 系列：旧版 ID 段
- 短 ID（如 `108_0001`、`175_0023`）：分组 ID + 子序号

宗门技能本身的 ID 编码（用户业务规约）：

```
30122001
│  │ │ │
│  │ │ └── 序号
│  │ └──── 五行：1=金 / 2=木 / 3=水 / 4=火 / 5=土
│  └────── 阶位：1=人阶 / 2=地阶 / 3=天阶
└───────── 类别：1=招式 / 2=奇术 / 3=神通 / 4=身法 / 5=传承心法
```

例如 `30122001` = `30 | 1（招式）| 2（地阶）| 2（木）| 001（序号）` = **木宗门 地阶 招式 第 1 个 = 坠叶三叠**。

> 注意：实际命名中也有 7 位 ID（如 `301115`），属于早期 ID 编码，不严格对齐 8 位规则。

## 关键易错点

1. **ID 重复**：跨文件出现相同 `SkillEffectConfig_<ID>` 节点会触发 `JsonGraphManager.CheckError`。新建技能必须从 `SvnIgnore/LocalID.json` 分配未使用 ID。
2. **TableTash 不一致**：表结构变更时 `CustomJsonConverter.TryCustomRead` 按字段名兜底；字段重命名而无兼容逻辑时丢值。
3. **节点拷贝重名**：`SkillTagsConfigNode.Enable` 检测 `createdFromDuplication` 后给 Desc 加 `_[ID]` 后缀，**保存前必须改回**否则 OnSaveCheck 拒绝。
4. **CdType=连招 时最后段 BaseDuration 必须为 0**：`SkillConfigNode.OnSaveCheck` 自动修正并报警。
5. **AI 标签"解控"必须开 SkillProperty 位**：解控技能必须配 `TBSP_IGNORE_SKILL_FORBIDDEN`。
6. **筛选默认敌军通用**：TSKILLSELECT_* 默认 `SpecialSkillSelectFlag = TSSSF_Common_DiJun`。配友军/特定单位需手动改。
7. **TSET_REPEAT_EXECUTE 间隔=0 + 次数>100 直接报错**：防死循环。
8. **TParam.ParamType 常见 5 不是百分比**：通常是"槽位/枚举类型化值"，需查 CSV 确认。
9. **RefConfigBaseNode 不能作为模板的第一个节点**：会报错——查找模板节点时第一个常规节点优先。
10. **JSON 出现 `.json.mine`**：SVN 冲突未解决，编辑器强制弹窗。

## 数据流转图

### 编辑器侧

```
策划在 GraphView 拉节点 / 连线
        ↓
GraphProcessor 框架内存图
        ↓ Save
SkillGraph 实例 → JsonGraphManager.SaveJson()
        ↓
SkillGraph_<ID>.json （磁盘）
        ↓
JsonGraphManager 缓存 (路径→GraphInfo)
        ↓
ExcelExport（导表）
        ↓
SkillConfig.xlsx / SkillEffectConfig.xlsx / ...
```

### 运行期侧

```
游戏启动 → TableDR_CS 加载 .bytes 表
        ↓
战斗发起 → 玩家释放技能 30122001
        ↓
SkillConfig 表查 ID → 取 SkillEffectExecuteInfo.SkillEffectConfigID
        ↓
SkillEffectConfig 表查 → 拿到 SkillEffectType + Params
        ↓
BattleSkillEffectProcessor switch (SkillEffectType)
        ↓ TSET_ORDER_EXECUTE
        ↓ → 遍历 Params, 对每个 ID 递归
        ↓
        ↓ TSET_CREATE_BULLET
        ↓ → 实例化 Bullet 单位
        ↓
        ↓ TSET_APPLY_ENTITY_EFFECT
        ↓ → 加视觉效果
       ...
```
