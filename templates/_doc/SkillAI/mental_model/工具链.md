## 一句话本质

`doc/SkillAI/tools/` 是 SkillAI 系统的工程实现层——把"IR YAML → SkillGraph JSON"双向转换、Lint、Mermaid 出图、ID 分配、Bootstrap 学习管线串起来的 Python 工具集。其中 `skill_compiler.py` 是**最复杂的工具**，编译时容易踩的 SkillEditor 内部规则全沉淀在这里。

## §A skill_compiler.py 7 个核心坑

> 来源：[memory/feedback_skill_compiler_pitfalls.md](../../../memory/feedback_skill_compiler_pitfalls.md) / PoC 阶段已踩坑全沉淀。所有 7 个坑都已在 skill_compiler.py 中以代码护栏形式固化。

### 坑 1：动态端口节点的 edges outputPortIdentifier 必须是 `"0"`

**规则**：xNode/GraphProcessor 框架中，`dynamicPortList` 风格节点（端口数量可变长）的所有出边 `outputPortIdentifier` 必须**全部为字符串 `"0"`**，而**不是** Params[N] 索引。

**因果链**：
- 编辑器加载 edges 时，对每个 edge 查节点的端口列表
- 动态节点只暴露一个聚合锚点端口 `"0"`
- 使用 `"1"` / `"2"` / ... 的边被丢弃 → 子节点视觉悬空（数据正常，看起来像没连）

**真实样本验证**：30122001 的 ORDER_EXECUTE 32000163 两条出边都是 `"0"`，但 `ConfigJson.Params=[32000162, 32000164]` 顺序正确——**Params 数组才是真相，edges 只是视觉**。

**已纳入代码护栏**：`skill_compiler.py` 的 `derive_edges()` 维护 `DYNAMIC_PORT_NODES` 集合（`TSET_ORDER_EXECUTE` / `TSET_NUM_MAX` / `TSET_NUM_MIN` / `AND` / `OR` 等），这些节点的所有 `PackedParamsOutput` 出边强制设 `outputPortIdentifier = "0"`。其他固定 schema 节点（DELAY/SWITCH/CREATE_BULLET 等）正常用 Params 索引。

详见 [SkillEditor文件结构.md §D 铁律 3](SkillEditor文件结构.md)。

### 坑 2：SkillConfig 必须补全 MOBA 通用默认值

**规则**：SkillConfigNode 的 ConfigJson 中以下字段不能留空（影响 AI 行为 + 编辑器显示）：

| 字段 | 默认值（从 30122001 提炼） | 用途 |
|------|--------------------------|------|
| `SkillIndicatorType` | 按 flow 中第一个 bullet → 2 直线 / aoe_circle → 3 圆形 / 否则 0 | 施法指示器类型 |
| `SkillIndicatorParam` | `[skill.range]` | 施法指示器参数 |
| `SmartCastTargetBasePriority` | 3（距离最近） | 智能施法 |
| `SmartCastTargetCondTemplate` / `CastTargetCondTemplate` | 1 | 目标条件模板 |
| `SmartCastTargetMonsterRankCond` / `CastTargetMonsterRankCond` | `[16,15,14,13,12,11,4,5,6,0]` | 怪物级别过滤 |
| `SmartCastTargetCampCond` / `CastTargetCampCond` | `[2]` 敌方 | 阵营过滤 |

**因果链**：留空 = 没目标可释放 / AI 不会用 / 编辑器面板大段空白。

**注意**：SkillIndicatorType 需要**递归扫描 delay.then / if.then 等嵌套层**，不是只看顶层。

**已纳入代码护栏**：`skill_compiler.py` 的 `make_skill_config_node()` + IR Schema 的 `skill.indicator/ai_cast/cast_target` 可选字段。

### 坑 3：引用类节点（BulletConfigNode/ModelConfigNode）必须显式生成

**规则**：TSET_CREATE_BULLET 的 BulletID 字段 / TSET_CREATE_EFFECT 的 ModelID 字段，在 SkillEditor 中**应该是连到一个独立的 BulletConfigNode/ModelConfigNode 节点**（拉出来的子弹/模型属性节点），不是只填一个 ID 数字。

**因果链**：策划需要**直观看到子弹/模型属性**（速度、射程、模型路径等）。否则只看到整数 ID 完全无法判断对错。

**节点 ConfigJson 内容**：从 `1SkillEditor.xlsx` 的 BulletConfig / ModelConfig sheet 按 ID 读取整行，inline 进 ConfigJson（约 27~49 个字段）。

**xlsx 数据陷阱**：
- xlsx 单元格里存的是**中文枚举字符串**（如 `FlyType="-"`、`AttachPos="空"`），编辑器把它反序列化为 int 会失败
- **对策**：编译器过滤掉中文枚举字符串字段（保留数字 / bool / 路径字符串 / 英文标识符），让 C# 反序列化用类默认值
- **xlsx 布尔陷阱**：表格里 bool 字段实际是文本 `"True"` / `"False"`，需要显式转换

**已纳入代码护栏**：`skill_compiler.py` 的 `make_bullet_config_node` / `make_model_config_node` + `_normalize_xlsx_value` + `_filter_safe_fields`。

详见 [SkillEditor文件结构.md §E RefConfigBaseNode 包装](SkillEditor文件结构.md)。

### 坑 4：子弹伤害链路必须通过模板组装（不能直接造伤害）

**规则**：MOBA 子弹的命中伤害**不能**用 TSET_DAMAGE_QUICK 直接造，必须走"碰撞模板 → 命中后 ORDER → 伤害模板 + 表现模板"标准链路。

**完整拓扑**（来自 30222005 真实样本）：

```
TSET_CREATE_BULLET
    └ Params[0] BulletID → BulletConfigNode
                            └ AfterBornSkillEffectExecuteInfo.SkillEffectConfigID
                                ↓
                            子弹通用碰撞模板 (TSET_RUN_SKILL_EFFECT_TEMPLATE)
                              路径: 技能模板/子弹/SkillGraph_【模板】子弹通用逻辑-碰撞.json
                              模板根 ID = 190016404
                              Params[3] 命中后功能 → ORDER
                                                      ├ 伤害模板（路径: 子弹通用逻辑-伤害.json, 根=190016485）
                                                      └ 表现模板（路径: 子弹通用逻辑-表现.json, 根=190016523）
```

**关键模板参数**：
- **碰撞模板**：`[命中后功能, 范围类型(1=圆/2=矩/3=扇/4=环), 半径, 高度, _, 偏移右, 偏移前, 目标阵营(0=敌军), _, 跟随创建者?, 侦测间隔, 侦测冷却, 侦测次数]`
- **伤害模板**：`[子弹序号, 伤害来源(entity:施法者伤害归属), 伤害类型(0=随技能), 伤害五行(0=随技能), 子类型 flags, 威力系数(-1=读参数), 额外伤害(-1=读参数), 增伤条件, 增伤形式, 增伤值, 命中后销毁, 护体灵光削减, 吟唱反制]`
- **表现模板**：`[自定义额外效果, 基础模式(0=命中目标才播), 命中特效模型 ModelConfig, 命中特效位置, 缩放%, 喷射强度, 顿帧, 受击震屏方向/强度/时长, 主角参与命中顿帧, 受击动作级别, 击退表现力度, 受击音效, 屏幕模糊时长, 波纹...]`

**TemplateData 必填**：TSET_RUN_SKILL_EFFECT_TEMPLATE 节点必须含 `TemplateData = {TemplateParams: [...], TemplatePath: "..."}`，TemplateParams 从目标模板 JSON 的根节点读取后复制（用 `load_template_params()`）。

**已纳入代码护栏**：`skill_compiler.py` 的 `KNOWN_TEMPLATES` 字典 + `make_template_call_node()` + `expand_bullet_straight()` 自动展开 hit 块。

详见 [伤害管线.md §A 6 层管道架构](伤害管线.md) + [子弹系统.md §E 子弹序号-威力 Tag 配对](子弹系统.md)。

### 坑 5：引用节点 ConfigJson 必须完整 — 优先从现有 JSON 拷贝

**规则**：BulletConfigNode/ModelConfigNode/BuffConfigNode 的 ConfigJson **必须包含全部字段**，特别是结构化 dict 字段：

```
- BeforeBornSkillEffectExecuteInfo: {SelectConfigID:0, SkillEffectConfigID:0}
- AfterBornSkillEffectExecuteInfo: {...}
- DieSkillEffectExecuteInfo: {...}
- LifeFlag / FlyType / AngleAdjustType 等中文枚举字段（编辑器保存为 int）
```

**症状**：编辑器加载能显示节点，但点"导出数据"时弹窗报错：

```
[ID异常] 模板效果【模板】子弹通用逻辑-表现
[SkillEffectConfig:0], ID:0, ConfigID:0
```

**修复（最稳路径）**：编译器优先从**项目内已有的 SkillGraph JSON** 中拷贝同一 `(cls, config_id)` 的 ConfigJson——编辑器保存过的数据 100% 合法、字段完整、类型正确。降级才查 xlsx。

**实现**：
- `_KNOWN_REF_NODE_CACHE`：扫 `Saves/Jsons/**/SkillGraph_*.json`（**跳过 `AIGen/`** 避免污染），按 `(cls, config_id)` 索引 ConfigJson
- `get_known_ref_config(cls, config_id)`：优先返回已知合法数据
- `make_bullet_config_node` / `make_model_config_node`：先查缓存，找不到才走 xlsx fallback

**已纳入代码护栏**：`skill_compiler.py` 的 `_KNOWN_REF_NODE_CACHE` + `_ensure_known_ref_cache()`。

### 坑 6：子弹伤害模板"伤害来源"参数 V=6（不是 V=4）

**规则**：`子弹通用逻辑-伤害` 模板 Params[4]「伤害来源」应该填 `entity:施法者伤害归属`（TCommonParamType=6 = TCPT_CREATE_SKILL_DAMAGE_PROPERTY_ORIGIN_ENTITY），**不是** `entity:主体伤害归属`（V=4）。

**因果链**：从 30222005 真实样本验证。子弹的"伤害来源"语义上是"施法者-伤害属性归属"，因为 MOBA 子弹是脱离施法者飞行的，伤害归属要追溯到施法者本人（与 [参数与上下文.md §嵌套主体语义陷阱](参数与上下文.md) 同源）。

**已纳入代码护栏**：`expand_bullet_straight()` 中 damage_node 的 Params[1]（= Params[4] of template call）已改为 `entity:施法者伤害归属`。

### 坑 7：Python bool 类型透出导致 ConfigJson 反序列化失败 → "ID异常"

**症状**：编辑器加载技能图能显示节点，但点"导出数据"弹窗报：

```
[ID异常] 模板效果(...) [SkillEffectConfig:0]，ID: 0, ConfigID: 0
```

**根因链**：
1. SkillEditor 的 `[ID异常]` 校验（`Assets/Thirds/NodeEditor/Base/ConfigEditor/Graphs/ConfigGraph.cs:217`）：`if (id == 0 || configID == 0 || id != configID)` 报错
2. `GetConfigID()` 从 `Config` 对象（ConfigJson 反序列化结果）的 ID 字段读
3. 若 ConfigJson 反序列化失败 → `Config = null` → `GetConfigID() = 0`
4. **失败原因**：`Params[N].Value` 字段在 C# 是 `int`，但 JSON 里出现了 `false` / `true` → Newtonsoft.Json 反序列化失败 → 整个 Config 为 null

**Python 陷阱**：`isinstance(False, int) == True`（bool 是 int 子类）。Python 字典中 bool 值序列化 `json.dumps` 默认输出 JSON `false` / `true` 而非 `0` / `1`。

**对症修复**：
1. `resolve_ref()` 中 `isinstance(ref, bool)` 检查必须**先于** `isinstance(ref, int)`
2. `TParam.to_dict()` 强制转 int：`int(value) if isinstance(value, bool) else value`
3. 任何 IR `extra_params` 中的字面量布尔（如 `True/False`）应统一改为 `1/0`

**验证手段**：编译完成后 grep 输出 JSON：

```bash
rg '"Value": false|"Value": true|"ParamType": false' output.json
# 应该 0 个匹配
```

**已纳入代码护栏**：`skill_compiler.py` 的 `TParam.to_dict()` + `resolve_ref()` 顺序修正。

### 衍生坑 8：SkillEditor 模板不需要 SkillConfigNode

**规则**：制作 SkillEditor 模板（`Saves/Jsons/技能模板/`）时，**根节点应该直接是 ORDER (IsTemplate=true)**，不需要 SkillConfigNode 外壳。

**因果链**：模板的语义是"被其他技能图通过 RUN_SKILL_EFFECT_TEMPLATE 调用的可复用片段"，不是独立技能。SkillConfigNode 用于普通技能的 SkillConfig 表行注册——模板没有 SkillConfig 对应行。

**注意**：旧模板（如 1860196 多向子弹）有 SkillConfigNode 是历史遗留，**新模板应该不带**。

**已纳入代码护栏**：`skill_lint.py` 的 E003 规则放宽——模板图（含 IsTemplate=true 节点）允许无 SkillConfigNode。

详见 [模板系统.md §模板根类型](模板系统.md)。

### 衍生坑 9：模板内不能有孤立分支（"表格缺失"报错根因之一）

**症状**：模板加载后某些 ORDER 节点报错"表格缺失 SkillEffectConfig:XXXXX"，对应 ID 节点其实存在但孤立。

**根因**：编辑器的"表格缺失"校验对所有节点的 Params 引用扫描，但只追溯"被根节点连通的子图"。**孤立分支**（自己有出边但没入边）的节点会被认为"未使用"，进而它们被引用的 ID 也认为"不存在于活跃图中"。

**对策**：编译/生成模板时必须保证**每个非根节点都至少有 1 条入边**。生成阶段 1 简化版时跳过未实现分支（如内层流程）的同时，**也要删除该分支创建的所有节点**——不能只跳过引用，节点本身也要删。

**验证手段**：编译后扫节点入边数：

```python
incoming = {guid: 0 for ...}
for e in edges: incoming[e['inputNodeGUID']] += 1
orphans = [n for n in nodes if incoming[n.guid]==0 and not n.is_template_root and n.cls!='SkillTagsConfigNode']
assert not orphans, f"孤立节点: {orphans}"
```

**已纳入代码护栏**：`build_fan_layered_template.py` 重构，未实现的分支不再生成节点。

与 [伤害管线.md §M7 孤岛节点](伤害管线.md) 的差异：孤岛节点是"模板演化遗物"，孤立分支是"生成时漏删"——两种"节点存在但不生效"的子模式。

### 衍生坑 10：不同表的 TableTash 不同 — 用错会反序列化失败

**症状**：节点显示存在，Lint 通过，但运行时**节点逻辑像没生效**（用默认值）。最隐蔽的 bug，因为没有任何错误提示。

**根因**：SkillEditor 每张表有独立的 TableTash（schema hash）。如果给 SkillConditionConfig 节点（如 TSCT_VALUE_COMPARE）用了 SkillEffectConfig 的 TableTash，编辑器反序列化时 hash 不匹配 → Config 反序列化为 null → 节点 GetValue() 返回默认值 → **行为悄悄错**。

**已知 TableTash 速查**（从真实样本提取）：

| 表 | TableTash |
|----|-----------|
| SkillEffectConfig | `0CFA05568A66FEA1DF3BA6FE40DB7080` |
| SkillConfig | `E50E65EC1A6EC53E59E9B3A412153E83`（30122001）/ `0C2C5D4A7DC9280429315C2EEAD44FDF`（1860196） |
| SkillTagsConfig | `6A8A6883BDFDA1411BB2461E65CB2D9B` |
| **SkillConditionConfig** | **`ED89F46EAB95F7ACF5C1911A5A375278`** |
| ModelConfig | `B1156BD6A276DA7A8E082421FCF0B7A9` |
| BulletConfig | `A4AEA1F4B1BD0FA9FD2F066BE902466F` |

**额外要点**：
- TSCT_VALUE_COMPARE 等条件节点必须有顶层 `SkillConditionType` 字段（而不是 `SkillEffectType`）
- Config2ID 前缀也必须是对应表名：`SkillConditionConfig_<id>` 不是 `SkillEffectConfig_<id>`
- 对应顶层字段：SkillEffectConfig→SkillEffectType / SkillConditionConfig→SkillConditionType / SkillSelectConfig→SkillSelectType

**已纳入代码护栏**：`build_fan_layered_template.py` 给 cmp 节点显式传 `table_tash=TASH_SKILL_CONDITION` + 删除误带的 SkillEffectType 顶层字段。

详见 [SkillEditor文件结构.md §反直觉点 4 TableTash](SkillEditor文件结构.md)。

## §E id_allocator.py

### 职责

全工程 SkillGraph ID 唯一分配，避免：
- 冲突（同 ID 不同 cls 在不同蓝图）
- 越界（`> int32 max = 2,147,483,647` / PostMortem #018）

### 工作机制

- 扫所有 SkillGraph_*.json 收集已用 ID
- 分配新 ID 时按段位号系（[mental_model README §rule_6](README.md)）避让
- 输出 `id_allocations/*.json` 留痕
