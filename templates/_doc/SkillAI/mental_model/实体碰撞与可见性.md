## 一句话本质

**实体碰撞与可见性子系统是位移 / 隐身 / 替身期间对实体物理碰撞和视觉表现的直接控制**——不通过 buff，而是用 `TSET_SET_ENTITY_COLLISION_ENABLE_STATE` / `TSET_APPLY_ENTITY_EFFECT` / `TSET_CREATE_ROLE` 等节点直接修改实体状态。"奇术 / 身法"在编辑器里实际是 3+ 个原语的组合（SET_COLL × N + ADD_FORCE + APPLY_ENTITY_EFFECT + CREATE_ROLE 等），不是单一节点。

---

## 关键不变量（违反就 100% 出问题）

1. **`TSET_SET_ENTITY_COLLISION_ENABLE_STATE` Params 字段**（B-002 3 样本归纳，未源码确认）：
   - `Param[0]` = 75（entity_uid 字面量，**与实体级 SkillTag 的 PT=0 不冲突**）
   - `Param[1]` = 碰撞对象类型（已观察 1 / 2 / 129，语义待源码确认）
   - `Param[2]` = enable（0=关 / 1=开）
   - `Param[3]` = 作用范围（0 / 1）

2. **`TSET_APPLY_ENTITY_EFFECT` Params 字段**：
   - `Param[0]` = 75（entity_uid 字面量）
   - `Param[1]` = 效果类型 ID（推测 5 = 透明 / 8 = 隐身雾，待源码 / 表格确认）
   - `Param[2]` = enable
   - `Param[5-6]` = duration / strength

3. **位移有 4 类模式**（B-003 D-302 修订，依据 ADD_FORCE 主语 + 是否需要关碰撞）：

   | 模式 | 特征 | 典型场景 | 样本 |
   |------|------|---------|------|
   | **A：自身位移 + 关碰撞** | 多 SET_COLL_OFF / ON 配合 ADD_FORCE | 释放方位移到目标点 → 必须关碰撞防穿模 | 30215001 / 30222002（部分）/ 30433001 |
   | **B：自身位移 + 不关碰撞** | ADD_FORCE × N + SET_COLL=0 | 释放方位移但碰撞保留（推测：低速 / 短距离 / 物理引擎自处理） | 30315001（4 ADD_FORCE + 0 SET_COLL，hold-out） |
   | **C：拉拽 / 击退敌人** | ADD_FORCE × N + SET_COLL=0 | ADD_FORCE 主语是敌人非己方 | 30325001（6 ADD_FORCE + 0 SET_COLL，B-003） |
   | **D：仅控制可见性 / 无位移** | APPLY_ENTITY_EFFECT 透明 + SET_COLL × N，无 ADD_FORCE | 隐身 / 雾效 / 透明化 | 30222002 部分场景 |

   **削弱后的不变量 3**：`SET_COLL_OFF / ON` 是**模式 A 标配**，**不是 ADD_FORCE 标配**——B/C 模式都有 ADD_FORCE 但 SET_COLL=0。设计/审核位移类技能时，先按 ADD_FORCE 主语 + 是否关碰撞判定属于哪一类，再选对应原语组合。

   **v0.3 旧表述**："位移技能标准模式"——COLL_OFF → ADD_FORCE → DELAY → COLL_ON"。**v0.4 修订**：该范式仅是模式 A，不能泛化到所有 ADD_FORCE 场景。

4. **隐身 / 替身机制不通过 buff** — `TSET_APPLY_ENTITY_EFFECT` 直接控制视觉透明 / 雾效；`TSET_CREATE_ROLE` 直接生成替身实体。"奇术 = buff 串联"是**只有部分实现**的归纳——见 [Buff系统.md](Buff系统.md) 反直觉 3。

## 反直觉点（容易踩的）

1. **30215001 奔岩突进是真位移，但用 buff + 位移组合，不是单纯位移** — flow 内 ADD_FORCE + SET_COLL × 2 + ADD_BUFF × 2，三机制并存。把"位移技能"想成"只有 ADD_FORCE"是错的。

2. **30433001 潜行水底是隐身 + 替身（CREATE_ROLE × 1）+ 位移，三机制叠加** — SET_COLL × 4 + APPLY × 2（透明 + 隐身雾）+ CREATE_ROLE 替身 + REMOVE_ENTITY_AI（替身禁 AI）。**单看技能名"潜行"完全无法预测这套复合机制**。

3. **位移 / 隐身机制在 flow 顶层可见，不在模板内部** — v0.1 / v0.2 一度推测"位移类全在模板内部"，B-002 D-206 反查发现 3/3 样本机制在 flow 顶层显式可见，模板系统主要封装控制流时间轴而非碰撞 / 可见性原语。

4. **"位移"在编辑器里实际是 3+ 个原语的组合** — 不是单一节点。设计 / 审核位移类技能时必须查 flow 内是否有 ADD_FORCE / SET_COLL / GET_POS_BY_ANGLE 等节点的协作链，缺一不可。

5. **Param[1] = 75 的 entity_uid 字面量与实体级 SkillTag 不混淆** — 实体级 SkillTag 是 `Param[1]=0 / PT=0`（[PostMortem #019](../postmortem/2026-05-08-019-entity-level-tag.md)）；本子系统的 SET_COLL / APPLY_ENTITY_EFFECT 用 75 作字面量寻址实体。是两套独立寻址。

## 与其他子系统的因果关系

- 实体碰撞 → **Buff 系统**（协作而非取代）：30215001 同时用 SET_COLL + ADD_FORCE + ADD_BUFF（三套机制并存）；位移期间常挂"超级盔甲"buff 防控制
- 实体碰撞 → **控制流子系统**（嵌套关系）：SET_COLL_OFF / ON 在 ORDER 链中按顺序触发；COLL_ON 通常在 DELAY 之后（位移完成后开启）
- 实体碰撞 → **位移 / 物理子系统（待建）**：ADD_FORCE / GET_POS_BY_ANGLE 是位移核心，与本子系统强耦合
- 实体碰撞 → **替身机制（暂存档）**：CREATE_ROLE / REMOVE_ENTITY_AI 在 30433001 首次完整观察，待 B-003+ 见第 2 个替身样本再独立建页
