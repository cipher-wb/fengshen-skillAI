## 一句话本质

`TCommonParamType (PT=5)` 是技能系统的**"代词系统"** —— 每个枚举值都是一个**上下文相对的实体引用**，而非绝对地址。
**在嵌套调用栈里（BulletConfig.BornSE / Buff Effect / RUN_SKILL_EFFECT 子 effect），"主体 V=1"指调用栈最近一层，不是玩家主角**。要拿到"最初施法的玩家主角"必须显式用 `V=35 施法者-根创建者`。

## 基础常量

### §战斗帧率铁律：30 帧 = 1 秒（v0.2 / 2026-05-13 memory 升级入库）

> 来源：[memory/feedback_frame_rate_rule.md](../../../memory/feedback_frame_rate_rule.md) / 用户 2026-05-13 在 note review 时立法（发现"240帧4秒"基于 60fps 错算，实际应为 8 秒基于 30fps）。

**铁律**：本项目战斗系统运行帧率为 **30fps**，**不是 60fps**。

**换算速查表**：

| 帧数 | 秒数 | 帧数 | 秒数 |
|------|------|------|------|
| 1 帧 | 1/30 秒 ≈ 33.3 ms | 90 帧 | 3 秒 |
| 15 帧 | 0.5 秒 | 120 帧 | 4 秒 |
| 30 帧 | **1 秒** | 150 帧 | 5 秒 |
| 45 帧 | 1.5 秒 | 180 帧 | 6 秒 |
| 60 帧 | 2 秒 | 210 帧 | 7 秒 |
| 75 帧 | 2.5 秒 | **240 帧** | **8 秒**（NOT 4 秒）|

**应用 / Sensor**：

- 写 Sticky Note / 注释 / 文档时，帧数换算一律 **÷ 30**
- 审核 note 时，看到帧数换算必须检查是否用 30fps 基数
- 生成技能蓝图时，所有"X 帧 = Y 秒"描述必须以 30fps 为准
- REPEAT_EXECUTE interval / count 配置时记得换算（[子弹系统.md §B](子弹系统.md)）

**反直觉点**：很多游戏引擎默认 60fps，AI 出图时容易按 60fps 写"240帧 = 4秒"——这是系统性错误，会让策划误判技能时间节奏。

## 关键不变量

### §嵌套主体语义陷阱 (D-038 candidate / B-DESIGNER-CHAIN-001 入库)

**主张**（candidate / 待 ≥3 跨子系统例累积后走用户拍板升正式 / 不走 AI 自决升正式 4-gate）：

> 在 `BulletConfig.BeforeBornSE / AfterBornSE / DieSE / TrackEntityNoTargetSE` /
> `BuffConfig.OnAttachEffect / OnTickEffect / OnRemoveEffect` /
> 任意 `RUN_SKILL_EFFECT` 节点的子 effect 等**嵌套 SkillEffect 调用栈**中，
> `TCPT_MAIN_ENTITY (V=1, PT=5) "主体单位实例ID"` 指**调用栈最近一层的主体**，**不是**"最初释放技能的玩家主角"：
>
> - 子弹 BulletConfig 的 BornSE 上下文 → 主体 = 子弹自身
> - Buff 的 OnTick/OnAttach Effect 上下文 → 主体 = Buff 携带者
> - RUN_SKILL_EFFECT 触发的子 effect 上下文 → 主体 = 上一层 effect 的主体

**要在嵌套上下文获取"最初施法的玩家主角"必须显式使用**：

| 优先级 | V 值 | TCommonParamType | 语义 |
|--------|------|------------------|------|
| ⭐ 1 | **V=35** | TCPT_CREATE_ROOT_CREATOR_ENTITY_ID | **施法者-根创建者**：整条调用链最初的 caster / 穿透多层 buff/嵌套技能/子弹 |
| 2 | V=37 | TCPT_MAIN_CREATOR_ENTITY_ID | 主体-创建者：仅穿透 1 层嵌套 |
| 3 | V=3 | TCPT_CREATE_ENTITY | 施法者实例ID：易被中间 buff 层覆盖 |
| ❌ | V=1 | TCPT_MAIN_ENTITY | 调用栈最近层主体（在嵌套上下文几乎永远不是玩家主角） |

**作用域**：

- 影响：`BulletConfig.{Before,After,Die}BornSkillEffectExecuteInfo` / `BulletConfig.TrackEntityNoTargetSkillEffectConfigID` / `BuffConfig.OnAttach/OnTick/OnRemove Effect` / 任意 `RUN_SKILL_EFFECT` 子 effect
- 不影响：`SkillConfig.SkillEffectExecuteInfo` 顶层（V=1 = 玩家主角 / 合法）/ `SkillConfig.SkillEffectPassiveExecuteInfo` 顶层（同上）

**证据**：

- L1 用户裁决 (2026-05-13)：用户原话 "32001000 里的【被跟随目标单位】不可以填【常用参数值 -1- 主体单位实例ID】... 应该填：常用参数值 35-施法者-根创建者"
- L1 源码：`Assets/Scripts/TableDR_CS/Hotfix/Gen/common.hotfix.cs:3777-3968` TCommonParamType enum 46 项完整定义 + Description
- L1 金标：`SkillGraph_280103_【狮妖】连线.json` BeforeBornSE 的 FOLLOW_ENTITY P[1] = V=37 / CREATE_BULLET P[4] = V=37（避开 V=1 印证嵌套陷阱）
- L2 用户实测：`SkillGraph_2500001` r1 (V=1) 错向 → r2 (V=35) Unity 跑通验证

**仍不确定 / 升正式 readiness**：

- gate_b_threshold = 1/3：累积 ≥3 跨子系统反例（BulletConfig + Buff + RUN_SKILL_EFFECT 各 1）才走用户拍板升格通道
- gate_g_cross_tool：升正式前必须跑 fs 真扫 SkillGraph 全库 BornSE / OnTickEffect 节点 V= 分布 / 验证 V=1 在嵌套上下文 0 真用

### §TCPT 角色变形矩阵 (D-039 candidate / B-DESIGNER-CHAIN-001 入库)

**主张**（candidate / 描述性归纳 / 待 fs 真扫 + 多形态金标后走用户拍板升正式）：

TCommonParamType enum 46 项里描述"实体引用"的核心子集呈现规律性的 **4 角色 × 4 变形 对称矩阵**：

| 角色 \ 变形 | 原始 | +伤害归属 | +根创建者 | +直接创建者 |
|-------------|------|-----------|-----------|-------------|
| 主体 (MAIN) | **1** | 4 | 33 | **37** |
| 目标 (TARGET) | 2 | 5 | 34 | 38 |
| 施法者 (CREATE) | 3 | 6 | **35** | 39 |
| Buff 来源 (BUFF_SOURCE) | — | 32 | 36 | 40 |

**说明**：

- 原始列：调用栈相对引用 / 会随嵌套漂移（V=1 陷阱根源）
- +伤害归属列：锁伤害归属字段（CREATE_BULLET P[10]=V=41 同源思路）
- **+根创建者列 (33/34/35/36)**：跨调用栈不漂移 / 永远指向"最初那次施法"
- +直接创建者列 (37/38/39/40)：穿透 1 层嵌套 / 比根创建者弱

完整 46 项 dump 见 [memory/reference_tcpt_enum_full.md](../../../../../../Users/Administrator/.claude/projects/F--DreamRivakes2-ClientPublish-DreamRivakes2-U3DProj/memory/reference_tcpt_enum_full.md)（不重复全文）。

### §ORIGIN / ROOT 系列跨调用栈锁定规律 (D-039 子段)

**主张**（candidate / 与 §TCPT 矩阵同源 / 描述性归纳）：

- `TCPT_ORIGIN_*` 系列（V=7/13/15/16/24/25/26/27/28/30/41/44/45/46）
- `TCPT_ROOT_*` 系列（V=33/34/35/36/43）

= **跨嵌套调用栈不漂移** / 永远指向"最初那次施法"

**推论 1**：`CREATE_BULLET P[10] = {V=41, PT=5}` 伤害归属铁律 = 这条规律的具体应用（不被嵌套 buff/bullet 层漂移）。两金标样本（280103 / 2500001）CREATE_BULLET P[10] 均为 V=41 → 跨阵营印证 ORIGIN 锁定规律的实战通用性。

**推论 2（与 D-038 的耦合）**：

- D-038 是规律的反面教材（V=1 = 原始系列 / 跨嵌套漂移）
- D-039 是规律的正面陈述（ROOT 列 + ORIGIN 系列 = 跨嵌套锁定）

## 反直觉点

1. **"主体"在嵌套上下文里是子弹/buff 自己** — SkillConfig 顶层语义被错误外推到 BulletConfig.BornSE 是配技能最常见的翻车（D-038 触发场景）
2. **V=33 vs V=35** — 都是"主体的根创建者"系列但 V=35 是"施法者-根创建者"专属字段 / 跨整条调用链最稳定 / 用户裁决证据明确
3. **配 ORIGIN 系列不需要 IF 判断** — V=41 等字段引擎自动从初始技能实例取 / 不被中间层覆盖
